# 12 Repo Glossary and Programming Terms

## Purpose of this document

Define programming and architecture terms as they specifically appear in this repo.

## Static generation

- Definition: HTML is produced ahead of time during the build instead of being generated by a live server on each request.
- In this repo: `astro.config.mjs` sets `output: 'static'`, and routes like `src/pages/resources/index.astro` render to plain files in `dist/`.
- See it in action: `astro.config.mjs`, `src/pages/weeks/[...slug].astro`

## Progressive enhancement

- Definition: the base page works as HTML first, and JavaScript adds richer behavior afterward.
- In this repo: glossary and flashcards render full markup first; scripts then add filtering or export behavior.
- See it in action: `src/pages/glossary/index.astro`, `src/scripts/glossary-page.js`

## Normalization

- Definition: converting raw data into a cleaner, more consistent shape for the rest of the app.
- In this repo: `src/lib/site-data.js` builds normalized `weeks`, `glossaryEntries`, and `flashcardEntries`.
- See it in action: `src/lib/site-data.js`

## Derived data

- Definition: data computed from source data instead of authored directly.
- In this repo: `phase_refs`, `week_refs`, and `day_refs` on glossary and flashcards are derived from day assignments.
- See it in action: `src/lib/site-data.js`

## Route contract

- Definition: the implicit agreement between a route’s markup, data payload, and enhancement script.
- In this repo: `week-data-json` plus `.js-day-card` selectors plus `src/scripts/week-page.js` form a route contract.
- See it in action: `src/pages/weeks/[...slug].astro`, `src/scripts/week-page.js`

## Module boundary

- Definition: the line that separates one file’s responsibility from another’s.
- In this repo: `progress-storage.js` persists state, while `progress-page.js` renders it.
- See it in action: `src/scripts/progress-storage.js`, `src/scripts/progress-page.js`

## State shape

- Definition: the structure of persisted or in-memory state.
- In this repo: progress state includes `completedDays`, `completedWeeks`, `blockedDays`, `artifactLinks`, and `weekReflections`.
- See it in action: `src/scripts/progress-storage.js`

## Persistence

- Definition: storing data so it survives page refreshes.
- In this repo: IndexedDB is the primary persistence layer, wrapped by `src/scripts/idb-kv.js`.
- See it in action: `src/scripts/idb-kv.js`, `src/scripts/notes-storage.js`

## Local-first

- Definition: the browser can own the user’s working state without needing a server round-trip.
- In this repo: notes and progress work offline and export locally.
- See it in action: `src/scripts/notes-storage.js`, `src/scripts/progress-storage.js`

## Content pipeline

- Definition: the path by which authored content becomes runtime-ready data.
- In this repo: JSON files are normalized in `src/lib/site-data.js`, validated by scripts, and then consumed by routes.
- See it in action: `src/lib/site-data.js`, `tools/upgrade-glossary-to-concept-model.mjs`

## Import/export flow

- Definition: a path for moving local state into or out of the app as files.
- In this repo: progress exports JSON; notes export both JSON and Markdown.
- See it in action: `src/scripts/progress-page.js`, `src/scripts/notes-page.js`

## Enhancement script

- Definition: a small browser module attached to one route that adds interactivity after load.
- In this repo: `src/scripts/flashcards-page.js` filters and exports cards on the Flashcards route.
- See it in action: `src/scripts/flashcards-page.js`

## Embedded JSON payload

- Definition: a `<script type="application/json">` block embedded in the HTML for the client script to parse.
- In this repo: routes embed `home-data-json`, `roadmap-data-json`, `week-data-json`, `flashcards-data-json`, `progress-data-json`, and `notes-data-json`.
- See it in action: `src/pages/index.astro`, `src/pages/weeks/[...slug].astro`

## DOM hook

- Definition: a class or dataset attribute that scripts rely on to find and control markup.
- In this repo: `.js-week-complete`, `.js-flashcard-item`, and `.js-notes-tab` are DOM hooks.
- See it in action: `src/pages/progress/index.astro`, `src/pages/flashcards/index.astro`, `src/pages/notes/index.astro`

## Usage metadata

- Definition: contextual tags added to content based on where it appears in the curriculum.
- In this repo: glossary terms and flashcards gain `phase_refs`, `week_refs`, and `day_refs`.
- See it in action: `src/lib/site-data.js`

## Object-store wrapper

- Definition: a thin abstraction over IndexedDB operations.
- In this repo: `src/scripts/idb-kv.js` exposes `getKV`, `setKV`, and `delKV`.
- See it in action: `src/scripts/idb-kv.js`

## Migration

- Definition: moving stored or authored data from one schema/shape to another.
- In this repo: `storage-migrate.js` migrates legacy localStorage data to IndexedDB, and `upgrade-glossary-to-concept-model.mjs` migrates glossary entries to a richer schema.
- See it in action: `src/scripts/storage-migrate.js`, `tools/upgrade-glossary-to-concept-model.mjs`

## Validation

- Definition: automated checks that reject malformed content or schema drift.
- In this repo: `scripts/validate_glossary_model.mjs` validates glossary structure, and `scripts/audit_flashcards.mjs` validates flashcard quality.
- See it in action: `scripts/validate_glossary_model.mjs`, `scripts/audit_flashcards.mjs`

## Build-time path generation

- Definition: generating route variants during the static build.
- In this repo: `getStaticPaths()` in `src/pages/weeks/[...slug].astro` emits both canonical and legacy slug paths.
- See it in action: `src/pages/weeks/[...slug].astro`

## Base-path awareness

- Definition: writing links and assets so they still work when the app is deployed under a subpath.
- In this repo: both `src/lib/site-data.js` and `src/scripts/runtime/client-utils.js` expose `withBase()`.
- See it in action: `src/lib/site-data.js`, `src/scripts/runtime/client-utils.js`

## Skill takeaway

A repo-specific glossary is useful because terms like “normalization,” “enhancement,” and “route contract” are easier to learn when tied to concrete files instead of abstract framework tutorials.
